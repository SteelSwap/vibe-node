# M6.10 — Conformance Gap Analysis Design

**Goal:** Systematic audit of every vibe-node subsystem against Cardano specs, Haskell reference code, and upstream issues/PRs/tags. Fix critical gaps, document the rest, validate on devnet.

**Branch:** `m6.10-conformance-gap-analysis`

---

## Audit Process (per subsystem)

Each work item follows the same methodology:

1. **Spec search** — query `vibe-node db search` for relevant formal spec sections (Shelley ledger spec, Ouroboros Praos paper, CDDL schemas, CIPs)
2. **Haskell code review** — read corresponding modules in `vendor/`, identify key functions, invariants, and edge case handling
3. **Upstream issue/PR/tag scan** — check GitHub issues, merged PRs, and recent release tags on upstream repos for bug fixes that could affect our implementation
4. **Gap identification** — compare our code vs spec + Haskell. Classify each finding:
   - **Missing feature** — spec/Haskell implements it, we don't
   - **Incorrect behavior** — our logic diverges from spec/Haskell
   - **Shared Haskell bug** — bug fixed in Haskell that our code is still prone to
   - **Encoding mismatch** — CBOR or wire format differs
5. **Severity classification:**
   - **Critical** — causes block rejection, chain divergence, or consensus failure
   - **High** — causes validation disagreement or protocol violation
   - **Medium** — functional difference unlikely to cause rejection but technically incorrect
   - **Low** — cosmetic, optimization, or edge case that doesn't affect correctness
6. **Fix critical/high** — implement fixes for anything causing block rejection, chain divergence, or security risk
7. **Document** — add entries to `docs/specs/gap-analysis.md` using the standard format from CLAUDE.md

### M6.10.7 — Devnet Regression Run

**Infrastructure:** `infra/devnet/docker-compose.devnet.yml`

**Procedure:**
- 3-node devnet (2 Haskell producers + 1 vibe-node producer)
- Run for 1000+ epochs (~6 hours at 0.2s slots, 100-slot epochs)
- Monitor: tip agreement, block acceptance rate, nonce consistency, fork switches
- Collect: vibe-node logs, Haskell node logs, block counts per producer

**Success criteria:**
- Zero VRF/nonce/validation errors
- Zero InvalidBlock rejections
- Tip agreement within k slots (10) for entire run
- Vibe-node block share 25–40% (expected 33%)

**Deliverable:** `docs/reference/conformance-report.md` with per-subsystem pass/fail
