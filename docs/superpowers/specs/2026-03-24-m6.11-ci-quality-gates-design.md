# M6.11 — CI Quality Gates Design

**Goal:** Establish code quality enforcement — formatting, linting, type checking, and contributor documentation.

**Branch:** `m6.11-ci-quality-gates`

---

## Components

### 1. Linting & Formatting Config (`pyproject.toml`)

- **ruff**: Rules E, F, I (isort), UP (pyupgrade), W. Line length 99. Exclude `vendor/`, `.venv/`, `build/`.
- **black**: Line length 99. Exclude `vendor/`, `.venv/`, `build/`.
- No mypy/ty initially — start with ruff + black, add type checking incrementally.

### 2. Pre-commit Hooks (`.pre-commit-config.yaml`)

- **ruff** — lint + isort on staged files
- **black** — format check on staged files
- Install via `pre-commit install`. Runs on `git commit`.

### 3. One-Time Formatting Pass

- Run `ruff check --fix` and `black .` on `src/` and `packages/` directories.
- Single commit: `style: apply ruff + black formatting to entire codebase`
- No logic changes — purely mechanical formatting to establish baseline.
- Going forward, only diffs are checked (pre-commit hooks + CI).

### 4. CI Lint Workflow (`.github/workflows/lint.yml`)

- Triggers on PR to main.
- Jobs: `ruff check`, `black --check`.
- Fails if code doesn't match configured formatting/lint rules.
- Runs on `ubuntu-latest` with Python 3.14.

### 5. Type Checking (`ty`)

- Start with `packages/vibe-core` (cleanest package).
- Expand to `vibe-cardano` incrementally.
- Configure strictness appropriate for current codebase state.
- Add to CI as non-blocking initially (warning only).

### 6. CONTRIBUTING.md

- Development setup — clone, `uv sync`, build VRF native
- Code style — ruff + black, dataclass conventions
- Branch naming — `m6.1-*`, etc.
- Commit message format — Co-Authored-By tag requirement (challenge rule)
- PR process — one branch per module, review before merge
- Test requirements — conformance tests against Haskell node
- NO worktree file copying — explicitly forbidden

### 7. Code Style Guide (`docs/reference/code-style.md`)

- Import ordering — stdlib, third-party, local (enforced by ruff isort)
- Type annotations — required on all public APIs
- Logging conventions — structured extra fields
- Error handling — custom exceptions, no bare except
- Dataclass usage — frozen + slots by default
- CBOR usage — `cbor2pure` not `cbor2` (buggy C bindings)

## Enforcement Model

- **Local:** Pre-commit hooks catch issues before commit
- **CI:** Lint workflow fails PR if violations exist
- **Baseline:** One-time format pass establishes clean state
- **Going forward:** Only diffs are checked — no retroactive enforcement on untouched code
