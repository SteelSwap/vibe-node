# M6.10 — Conformance Gap Analysis Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Systematically audit every vibe-node subsystem against Cardano specs, Haskell reference code, and upstream issues/PRs/tags. Fix critical gaps, document all findings, validate on devnet.

**Architecture:** 6 subsystem audits (consensus, networking, ledger, storage, serialization, plutus) each following the same methodology: spec search, Haskell code review, upstream issue/PR/tag scan, gap identification, fix critical/high, document. Followed by a 1000+ epoch devnet regression run.

**Tech Stack:** `vibe-node db search` (spec database), `vendor/` Haskell submodules, GitHub API (`gh`), devnet Docker Compose

**Spec:** `docs/superpowers/specs/2026-03-24-m6.10-conformance-gap-analysis-design.md`

## Audit Methodology (applies to Tasks 1-6)

Each subsystem audit follows these steps:

1. **Spec search** — `vibe-node db search "<subsystem keyword>"` for relevant formal spec sections
2. **Haskell code review** — read key modules in `vendor/<repo>/`, identify invariants and edge cases
3. **Upstream scan** — use `gh` CLI to search issues, PRs, and tags on upstream repos:
   - `gh issue list -R IntersectMBO/<repo> --search "<keyword>" --state all --limit 50`
   - `gh pr list -R IntersectMBO/<repo> --search "<keyword> fix" --state merged --limit 50`
   - `gh release list -R IntersectMBO/<repo> --limit 10`
   - Check release notes for bug fixes relevant to our implementation
4. **Gap identification** — for each finding, classify:
   - Severity: critical / high / medium / low
   - Type: missing feature / incorrect behavior / shared Haskell bug / encoding mismatch
5. **Fix** — implement fixes for critical and high severity gaps
6. **Document** — add entries to `docs/specs/gap-analysis.md` using the standard format:

```markdown
## [Subsystem] — [Brief description]

**Spec reference:** [Document, section, page/equation number]
**Era:** [Which era]
**Spec says:** [What the spec defines]
**Haskell does:** [What Haskell implements]
**Delta:** [The specific difference]
**Implications:** [How this affects our implementation]
**Severity:** [critical/high/medium/low]
**Status:** [fixed/documented/deferred]
**Discovered during:** M6.10
```

7. **Test** — run existing test suite after any fixes: `uv run pytest -x -q`


Monitor periodically:
```bash
docker logs devnet-vibe-node-1 2>&1 | grep -c "Forged block"
docker logs devnet-vibe-node-1 2>&1 | grep -c "Epoch transition"
docker logs devnet-haskell-node-1-1 2>&1 | grep -i "VRF\|BadProof\|Invalid\|UnexpectedPrev" | grep -v config
```

- [ ] **Step 4: Collect final statistics**

After 1000+ epochs (~6 hours):
```bash
FORGED=$(docker logs devnet-vibe-node-1 2>&1 | grep "Forged block" | wc -l)
EPOCHS=$(docker logs devnet-vibe-node-1 2>&1 | grep "Epoch transition" | wc -l)
EXTENDS=$(docker logs devnet-haskell-node-1-1 2>&1 | grep "AddedToCurrentChain" | wc -l)
VRF_ERR=$(docker logs devnet-haskell-node-1-1 2>&1 | grep -i "VRFKeyBadProof\|BadVRFValue" | wc -l)
INVALID=$(docker logs devnet-haskell-node-1-1 2>&1 | grep -i "InvalidBlock" | wc -l)
echo "Forged: $FORGED, Epochs: $EPOCHS, Total chain: $EXTENDS, VRF errors: $VRF_ERR, Invalid: $INVALID"
```

**Success criteria:**
- Zero VRF/nonce/validation errors
- Zero InvalidBlock rejections
- Tip agreement within k=10 slots for entire run
- Vibe-node block share 25–40% (expected 33%)
- 1000+ epoch transitions
