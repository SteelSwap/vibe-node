## Values in Plutus Core {#sec:uplc-values}

The semantics of built-in functions in Plutus Core are obtained by instantiating the sets $\Con{\tn}$ of constants of type $\tn$ (see Section [\[sec:builtin-inputs\]](#sec:builtin-inputs){reference-type="ref" reference="sec:builtin-inputs"}) to be the expressions of the form `(con` $\tn$ $c$`)` and the set $\Inputs$ to be the set of Plutus Core *values*, terms which cannot immediately undergo any further reduction, such as lambda terms and delayed terms. Values also include partial applications of built-in functions such as `[(builtin modInteger) (con integer 5)]`, which cannot perform any computation until a second integer argument is supplied. However, partial applications must also be *well-formed*, in the sense that applications of `force` must be correctly interleaved with genuine arguments, and the arguments must themselves be values.

We define syntactic classes $V$ of Plutus Core values and $A$ of partial builtin applications simultaneously:

::: minipage
$$\begin{array}{lrcl}
        \textrm{Value}  & V   & ::= & \con{\tn}{c} \\
                        &     &     & \delay{M} \\
                        &     &     & \lamU{x}{M} \\
                        &     &     & \constr{i}{\repetition{V}} \\
                        &     &     & A
    \end{array}$$ []{#fig:untyped-cek-values label="fig:untyped-cek-values"}
:::

Here $A$ is the class of well-formed partial applications, and to define this we first define a class of possibly ill-formed iterated applications $\pba$ for each built-in function $b \in \Fun$:

::: minipage
$$\begin{array}{lrl}
  \pba & ::= & \builtin{b}\\
       &     & \appU{\pba}{V}\\
       &    & \force{\pba}\\
    \end{array}$$ []{#fig:partial-applications label="fig:partial-applications"}
:::

We let $\pbas$ denote the set of terms generated by the grammar in Figure [\[fig:partial-applications\]](#fig:partial-applications){reference-type="ref" reference="fig:partial-applications"} and we define a function $\beta$ which extracts the name of the built-in function occurring in a term in $\pbas$: $$\begin{array}{ll}
 \beta(\builtin{b}) &= b\\
 \beta(\appU{\pba}{V}) & =\beta(\pba)\\
 \beta({\force{\pba}}) & =\beta(\pba)\\
\end{array}$$

We also define a function $\pbasize{\cdot}$ which measures the size of a term $\pba \in \pbas$: $$\begin{array}{ll}
\pbasize{\texttt{(builtin $b$)}} &= 0\\
\pbasize{\texttt{[$\pba$ $V$]}} &= 1+\pbasize{\pba}\\
\pbasize{\texttt{(force $\pba$)}} & = 1+\pbasize{\pba}
\end{array}$$

#### Well-formed partial applications.

A term $\pba \in \pbas$ is an application of $b = \beta(\pba)$ to a number of values in $S$, interleaved with applications of $\texttt{force}$. We now define what it means for $\pba$ to be a *well-formed partial application*. Suppose that $\alpha(b) =
[\iota_1, \ldots, \iota_n]$. Firstly we require that $\pbasize{\pba} < n$, so that $b$ is not fully applied; in this case we put $\iota=\iota_{\pbasize{\pba}}$, the element of $b$'s signature which describes what kind of "argument" $b$ currently expects. The definition is completed by induction on the structure of $\pba$:

1.  $\pba=\mathtt{(builtin}\ b \mathtt{)}$ is always well formed.

2.  $\pba=\mathtt{[}\pba'\ V\mathtt{]}$ is well formed if $\pba'$ is well formed and $\iota \in \Unihash$ or $\iota = \star$ (equivalently, $\iota \notin \QVar$).

3.  $\pba=\mathtt{(force}\ \pba'\mathtt{)}$ is well formed if $\pba'$ is well formed and $\iota \in \QVar$.

The definition of values in Figure [\[fig:untyped-cek-values\]](#fig:untyped-cek-values){reference-type="ref" reference="fig:untyped-cek-values"} is now completed by defining $A$ to be the syntactic class of well-formed *partial* built-in function applications: $$A = \{\pba \in \pbas: \pba \text{ is a well-formed partial application} \}.$$

Note that this definition does not impose any requirements of type correctness. For example, with the types and functions defined in Section [\[sec:default-builtins-1\]](#sec:default-builtins-1){reference-type="ref" reference="sec:default-builtins-1"} the term $X =\texttt{[(builtin
    modInteger) (con string "blue")]}$ is a valid value which could be treated like any other, for instance by being passed as an argument to a `lam` expression. However, the evaluation rules described in the next section require that when a built-in function $b$ becomes *fully* applied the types of the arguments are checked against the signature of $b$ using the relation $\approx$ and the function $\Eval$ defined in Sections [\[sec:compatibility\]](#sec:compatibility){reference-type="ref" reference="sec:compatibility"} and [\[sec:eval\]](#sec:eval){reference-type="ref" reference="sec:eval"}, so an error would arise if the term $X$ were ever applied to another argument.
